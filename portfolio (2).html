Backend Overview

The gym website has a real, functioning backend — not just a static contact form. It uses Supabase as the backend platform, providing a hosted PostgreSQL database, serverless Edge Functions, and a REST API — all without managing a dedicated server.

The system is designed to capture every membership inquiry automatically, store it securely, and notify the gym owner in real time via WhatsApp — turning the contact form into a live lead management pipeline.

Backend Technologies Used
  • Supabase — hosted backend platform (database + functions + REST API)
  • PostgreSQL — relational database for persistent, structured lead storage
  • Supabase Edge Functions — serverless Deno runtime for server-side form handling
  • JavaScript Fetch API — asynchronous frontend-to-backend communication
  • WhatsApp deep link — instant owner notification on every form submission
  • Bolt.new — admin dashboard for viewing and managing all collected leads
Database Architecture

A PostgreSQL database hosted on Supabase stores every membership inquiry. The main table is leads.

ColumnTypeDescription

Why PostgreSQL?

PostgreSQL gives the gym owner a structured, queryable record of every lead. The admin dashboard (built on Bolt.new) connects directly to this table, allowing the owner to filter by plan, sort by date, and track follow-up status.

API Workflow — Step by Step

When a visitor submits the membership inquiry form, the following sequence runs automatically:

1
Visitor fills in name, phone, plan, and optional email/message
↓
2
JavaScript validates client-side — phone format, name length, email pattern
↓
3
Honeypot field is checked — bot submissions are silently discarded
↓
4
A POST request is sent via fetch() to the Supabase Edge Function endpoint
↓
5
Edge Function re-validates data server-side and inserts a row into the leads table
↓
6
Success response is returned — the form is replaced with a confirmation message
↓
7
Gym owner receives an instant WhatsApp notification with the lead's full details

Supabase Edge Function — submit-lead

A serverless Supabase Edge Function handles all form processing. It runs on Deno at the edge, close to the user, ensuring fast response times with no cold starts.

Responsibilities
  • Receive and parse the JSON form payload from the frontend
  • Perform server-side validation — phone format, required fields, data types
  • Check the honeypot field and reject bot submissions before any DB writes
  • Insert the validated lead record into the PostgreSQL leads table
  • Return a structured JSON success or error response to the frontend
Pseudo-code Logic
// Edge Function: submit-lead (Deno runtime)
if (honeypot !== '') return error('Bot detected');
if (!isValidPhone(phone)) return error('Invalid phone');
if (!name || !plan) return error('Missing required fields');

await supabase
  .from('leads')
  .insert({ full_name, phone, email, interested_in, message });

return { success: true, message: 'Lead saved successfully' };

Admin Dashboard

The backend integrates with an admin dashboard built using Bolt.new. It connects directly to the Supabase PostgreSQL database, giving the gym owner a clean interface to manage all leads:

  • View all submitted membership inquiries in a sortable table
  • Filter leads by membership plan (Basic, Standard, Premium)
  • Sort by submission date to prioritize the most recent inquiries
  • Track follow-up status for each potential member
